🚀 Quick Start

Step 1 — Create Devices TopoAssist > Topology Manager > Device Manager

Enter device names like Leaf1 or ranges like Leaf[1-4] (zero-padded ranges like Leaf[01-04] are also supported). Select the device type:

  • Arista (full) — gets all attribute columns, config is generated.
  • Non-Arista — gets only the int_ column (traffic generators, non-Arista switches). Devices assigned the IXIA or SPIRENT role are automatically Non-EOS. Any device can also be toggled manually by clicking the EOS badge in the device list — it switches to Non-EOS and back without requiring a delete/re-add. Changes take effect after Save + Force Sync.

Use the 👁 visibility toggle to hide devices from the topology without deleting their data. Drag rows to reorder devices (affects sheet column order).

Step 2 — Connect Interfaces
Option A: Spreadsheet Matrix (Fastest)
Find the column for the Remote Device (e.g. int_Leaf2). In the row for the Local Device (e.g. Leaf1), type the port name (e.g. Et1). The topology refreshes automatically.
Option B: Visual Edit Mode
Enable Edit Mode in the visualizer toolbar. Click a red dot (disconnected port) on one device, then click a red dot on another device to cable them. Use (+) Link Pair to add a new row.
Step 3 — Set Modes & Attributes

Fill in the sp_mode_ column for each port. This drives config generation and visual color-coding.

l2-et-access l2-po-access l2-et-trunk l2-po-trunk l3-et-int l3-po-int l3-et-sub-int l3-po-sub-int
Step 4 — Generate & Copy Config

Click any cable or device in the topology. A Copy Strip appears at the bottom of the toolbar with split buttons for the Device, Left endpoint, and Right endpoint. Click Copy to send EOS config to clipboard, or View to open a scrollable config preview.

⚙️ Configuration Generation

1. Interface Naming

The system validates and normalises Arista EOS interface names:

  • Et — Ethernet  |  Po — Port-Channel  |  Vl — Vlan SVI
  • Lo — Loopback  |  Ma — Management

2. MLAG Detection (Strict Count = 4)

A Port-Channel is automatically classified as MLAG if it appears on 4 or more rows across the full sheet. Physical links between the two MLAG peers are tagged as Peer-Links automatically.

Guard: MLAG + any L3 mode (l3-*) is invalid on Arista EOS. The config generator outputs an explicit !! ERROR comment block instead of broken config.

3. IP Addressing (VLAN-Derived)

Middle octets are derived from the VLAN ID:

  • VLAN 1050 → octets .10.50
  • Last octet = Device DID (sequential Arista device index, set in Device Manager).
IPv6 uses the same pattern as IPv4 for lab convenience — this intentionally deviates from strict Arista EOS syntax.

4. Auto-Description Logic

Descriptions are generated automatically unless overridden by a manual desc_ value:

  • Physical standalone: -> PeerDev-PeerPort
  • Physical member of Po: -> PeerDev-PeerPort-PoX
  • Peer-link physical: -> PeerDev-PeerPort-MLAG-PEER-LINK
  • Port-Channel (regular): -> PeerDev or -> PeerDev & PeerDev2
  • MLAG Port-Channel: -> PeerDev & PeerDev2 (MLAG)

5. Global Config Template

TopoAssist > Topology Manager > Global Config

Define a base config template (NTP, management, AAA, etc.) that is prepended to every device's generated output.

6. Technology Stack

Configure interface types and routing protocols via the toolbar button. These flags control which config blocks are included in the generated output.

  • Gateway (top) — two independent checkboxes: IPv4 (ip address + VARP) and IPv6 (ipv6 address + VARP) on SVI, sub-interface, and L3 routed ports. Both default on.
  • P2P Interface (5-mode radio) — sets the addressing mode for point-to-point underlay links:
    • IPv4 — ip address only
    • IPv6 — ipv6 address only (explicit, no IPv4)
    • IPv6 Unnum — ipv6 enable + ipv6 unnumbered Loopback0 (no IPv4)
    • IPv4 + IPv6 — both ip address and ipv6 address
    • IPv4 + IPv6 Unnum — ip address + ipv6 unnumbered Loopback0
  • Protocol grid (4 columns) — checkboxes for BGP, OSPF, VXLAN, and EVPN variants. Options that are incompatible with the selected P2P mode are greyed out and automatically unchecked.
  • Validation errors — the modal blocks Save and shows an error banner if an invalid combination is chosen (e.g. IPv6 GW with no IPv6 routing protocol, VXLAN without an underlay, EVPN without BGP).

7. Snake Test (L2 / L3)

A snake test chains switch ports with external loopback cables to measure forwarding throughput end-to-end. A traffic generator (IXIA, Spirent, etc.) transmits frames on the TRAFFIC_IN port; the VRF chain routes traffic through every loopback cable pair until it exits on the TRAFFIC_OUT port where the generator measures what was received.

Supported Modes

Modesp_modeip_typeWhat is generated
L2 trunkl2-et-trunk(none)switchport trunk + allowed vlan on both ports. No routing, no VRF.
L3 routedl3-et-intp2pno switchport + per-port VRF (SNAKE_EtX) + .1/24 IP + static ARP + egress-vrf route chain.
L3 sub-intl3-et-sub-intp2pno switchport on parent + EtX.VLAN sub-interface per port in its own VRF + same ARP/route chain.

Physical Setup

  • Cable each snake primary ↔ secondary pair with a direct loopback patch cable (e.g., Et2 ↔ Et3, Et4 ↔ Et5, …).
  • Connect the traffic generator's TX port to the TRAFFIC_IN switch port (a regular P2P row, separate from the snake pairs).
  • Connect the traffic generator's RX port to the TRAFFIC_OUT switch port (another regular P2P row).
  • Each snake pair row needs a unique VLAN — the VLAN determines the L3 subnet ({base}.⌊vlan÷100⌋.vlan%100.0/24). The audit engine enforces uniqueness.

Sheet Setup — Snake Pairs

Each snake pair occupies one row. Open the Edit modal with the same device on both sides (devA = devB) — TopoAssist detects this and writes to both int_ and snake_int_ columns automatically. You can also fill the sheet directly:

  • int_Leaf1 = primary port (e.g., Et2)
  • snake_int_Leaf1 = secondary port (e.g., Et3)
  • sp_mode_Leaf1, vlan_Leaf1, ip_type_Leaf1 — set once per row, applied to both ports

Snake pairs appear as orange arcs looping off the right side of the device node on the canvas.

Sheet Setup — TRAFFIC_IN / TRAFFIC_OUT Ports

The two ports connected to the traffic generator are regular P2P rows (one device on each side, ip_type=p2p). TopoAssist identifies them by the desc_ column value:

desc_ valueRoleVRF assigned
TRAFFIC_IN_L3Traffic generator TX → switch RXSNAKE_{firstPrimary} — same VRF as the first snake pair's primary port
TRAFFIC_OUT_L3Switch TX → traffic generator RXSNAKE_{lastSecondary} — same VRF as the last snake pair's secondary port

By sharing the VRF with the first/last snake port, incoming traffic enters the routing chain directly and outgoing traffic leaves through the correct port. If either description is missing and snake pairs are present, the config viewer flags it in the 080_SNAKE block status.

Auto Config — Snake Test Fields

Open Auto Config → Snake Test to configure these fields:

FieldWhat it meansWhere used

IXIA MACMAC address of the traffic generator's RX (TRAFFIC_OUT) port.Static ARP on the last secondary VRF — arp vrf SNAKE_{lastSecondary} {ixia_nh} {ixia_mac} arpa.

Must be in the same /24 subnet as the TRAFFIC_OUT port's IP (e.g., if TRAFFIC_OUT is 200.15.0.1/24, this is 200.15.0.2).Next-hop for the last secondary static route — ip route vrf SNAKE_{lastSecondary} {dest} {ixia_nh}. When snake pairs are detected but any of these fields is empty, a yellow warning banner appears in the Auto Config modal listing exactly what is missing.

L3 VRF Chain — What Gets Generated

For each L3 snake pair, TopoAssist auto-generates across four config sections:

  • 001_SYSTEM — vrf instance SNAKE_Et2 + ip routing vrf SNAKE_Et2 (one entry per port; no ipv6 unicast-routing in snake VRFs).
  • 050_EtX — interface block per port: vrf SNAKE_EtX, ip address {base}.oct2.oct3.1/24, no shutdown. No OSPF (cannot form adjacency with yourself). Both the primary and secondary get their own 050_ block.
  • 050_TRAFFIC_IN / 050_TRAFFIC_OUT — regular P2P interface blocks, but in the first/last snake VRF respectively (auto-assigned from desc_).
  • 080_SNAKE — static ARP + route chain. Two ARP entries per pair (primary and secondary VRFs both map .2 to bridge MAC). Primary routes forward; intermediate secondaries use egress-vrf to cross into the next pair; last secondary routes to the traffic generator via IXIA MAC + Traffic Port P2P IP.

! Snake VRF Chain - Static ARP + Routing ! ! Pair 1: Et2 <-> Et3 (VLAN 200, subnet 200.2.0.0/24) arp vrf SNAKE_Et2 200.2.0.2 001c.7300.abcd arpa arp vrf SNAKE_Et3 200.2.0.2 001c.7300.abcd arpa ip route vrf SNAKE_Et2 10.99.99.98/32 200.2.0.2 ip route vrf SNAKE_Et3 10.99.99.98/32 egress-vrf SNAKE_Et4 200.4.0.2 ! ! Pair 2: Et4 <-> Et5 (VLAN 400, subnet 200.4.0.0/24) arp vrf SNAKE_Et4 200.4.0.2 001c.7300.abcd arpa arp vrf SNAKE_Et5 200.4.0.2 001c.7300.abcd arpa ip route vrf SNAKE_Et4 10.99.99.98/32 200.4.0.2 arp vrf SNAKE_Et5 200.15.0.2 0000.aaaa.bbbb arpa ← IXIA MAC at Traffic Port P2P IP ip route vrf SNAKE_Et5 10.99.99.98/32 200.15.0.2 ← next-hop exits via TRAFFIC_OUT port !

End-to-End Traffic Path (L3)

  1. Traffic generator sends frames destined to {ixia_subnet}.98/32 (e.g. 10.99.99.98/32) into the TRAFFIC_IN port.
  2. TRAFFIC_IN is in VRF SNAKE_Et2 (first primary's VRF). The static route points to 200.2.0.2; static ARP maps that IP to the switch's bridge MAC → frame exits Et2 with DMAC = bridge MAC.
  3. Loopback cable carries the frame to Et3. The switch accepts it (DMAC = own MAC).
  4. In VRF SNAKE_Et3, the route uses egress-vrf SNAKE_Et4 → frame exits Et4.
  5. Loopback cable to Et5. In VRF SNAKE_Et5, static ARP maps the traffic generator's port IP (200.15.0.2) to IXIA MAC → frame exits TRAFFIC_OUT toward the generator.
  6. Traffic generator receives the frame on its RX port and measures throughput / loss.

Bidirectional Traffic

The L3 VRF chain is inherently unidirectional: routing goes TRAFFIC_IN → snake chain → TRAFFIC_OUT only. Each loopback cable is used for full-duplex traffic at the hardware layer, but the IP routing only establishes one logical direction.

Most traffic generator applications handle bidirectional measurement by running two independent unidirectional streams simultaneously and aggregating the results. For a reverse direction through the same switch, you would need a second, independent snake chain wired in the opposite direction.

Manual Steps Required

  1. Loopback cables — one patch cable per snake pair (Et2↔Et3, Et4↔Et5, …).
  2. Bridge MAC — set in Auto Config → Snake Test. Auto-detected if Device Bridge is running; otherwise read from EOS with show version | grep "System MAC".
  3. IXIA MAC — MAC of the traffic generator's RX (TRAFFIC_OUT-side) port interface.
  4. Traffic Port P2P IP — IP of the traffic generator's interface on the TRAFFIC_OUT cable. Must be in the same subnet as TRAFFIC_OUT's ip address (e.g. if TRAFFIC_OUT has 200.15.0.1/24, set this to 200.15.0.2).
  5. IXIA Subnet — three-octet prefix of the test destination (e.g. 10.99.99). Configure the traffic generator to send traffic to {prefix}.98/32.
  6. TRAFFIC_IN / TRAFFIC_OUT descriptions — set desc_=TRAFFIC_IN_L3 on the port row connected to the generator's TX, and desc_=TRAFFIC_OUT_L3 on the port connected to the generator's RX.

Everything else — VRF declarations, interface configs, ARP entries, and the full routing chain — is generated automatically from the sheet data.

Audit Engine

Click the Audit icon in the toolbar to run topology integrity checks. A badge shows the error count when issues are found.

  • Po Mismatch — both sides of a cable have po_ set but to different Port-Channel numbers. Both ports are flagged.
  • Po Asymmetry — one side of a cable has po_ set and the peer does not. Both ports are flagged.
  • MLAG Destination Conflict — the same Po number points to different remote device names across links.
  • VLAN Mismatch — the vlan_ values differ across a cable (Arista-only links).
  • Mode Incompatible — the sp_mode_ pair across a cable is not a supported combination.
  • Invalid Interface Name — int_ does not match a standard EOS prefix (Et, Po, Lo, Vl, etc.).
  • Missing Po for Port-Channel mode — interface mode is po-based but po_ is blank.
  • Stale Po value on Ethernet mode — interface mode is et-based but po_ has a value.
  • Duplicate Interface — the same int_ name appears more than once for the same device.
  • Speed Mismatch — both sides of a cable must have the same speed_ value.
  • VLAN Gateway Conflict — the same VLAN ID is assigned different gateway IPs across devices.

Clicking a flagged port in the topology highlights it. Click the Audit icon again to toggle the mode off.

🔌 Device Bridge

The Device Bridge is a lightweight local Python server (device_bridge.py) that gives TopoAssist live read access to lab devices via SSH, eAPI, REST, or gNMI — without any cloud dependency. Its primary purpose is to reduce lab bring-up time by surfacing cabling errors, device identity issues, and interface state directly in the topology view.

Setup

Step 1 — Get the bridge script

Click the Device Bridge toolbar button (shield icon), then click Download Bridge in the modal footer. This saves device_bridge.py to your Downloads folder — it is always in sync with the current sidebar version.

Step 2 — Configure & run

Open device_bridge.py in a text editor and set the config at the top:

  • METHOD — transport to use: "ssh" (default), "eapi", "rest", or "gnmi"
  • JUMP_HOST — SSH jump server hostname (e.g. "bus-home"); set "" to SSH devices directly
  • SSH_USER — device login username (default "admin", passwordless)

Then run it in a terminal and keep it open:

python3 device_bridge.py

The bridge listens on http://localhost:8765. The modal shows a green dot when it is reachable.

Transport options:
ssh — CLI over SSH; requires OpenSSH + sshpass on the bridge machine. No pip deps.
eapi — Arista eAPI JSON-RPC over HTTPS (port 443); same output as SSH; single request per device. No pip deps.
rest — RESTCONF over HTTPS; OpenConfig YANG format. No pip deps. Requires EOS 4.22+.
gnmi — gRPC/gNMI; OpenConfig YANG format. Requires pip install pygnmi. Port 6030. EOS 4.22+.

Running a Check

Open the modal, enter or confirm the Host / IP for each device (defaults to device name for DNS-resolvable hosts), then click Run Checks. The bridge queries all devices in parallel. A spinner shows progress and updates as each sweep completes (LLDP done → LLDP + status done).

Device List — Status Dots

Each device row shows two indicator dots after the IP field:

SSH dot — green: bridge reached the device successfully.
SSH dot — red: SSH failed (wrong IP, auth issue, device unreachable).
SSH dot — gray: no IP configured for this device.
Hostname dot — green: device hostname matches topology name. Hover for confirmed hostname.
Hostname dot — amber: hostname mismatch — wrong device on that IP. Hover to see expected vs. actual name.
An amber hostname dot is a critical lab bring-up signal — it means the device responding on that IP is not the device you think it is. Check physical cabling and IP assignment before pushing config.

LLDP Results

Results are shown per cable, sorted by severity:

✓ Match — LLDP confirms the cable: both sides see the correct neighbor and port.
✗ Mismatch — LLDP sees a different neighbor or port than the topology expects. Detail line shows what was actually seen. Always flagged in Audit mode.
? Partial — one side was queried OK, the other was not (wrong IP or SSH failure on that device).
? Unknown — neither device was reachable; result is inconclusive.
— Skip — non-Arista device or non-Ethernet interface (LLDP check only covers Et↔Et Arista links).

Topology Card Indicators

After a check, results appear directly on topology device cards — no need to keep the modal open:

  • EOS version line — shown under the device hostname (e.g. EOS 4.32.1F · 7050SX3-48YC12). Populated from show version.
  • LL column — per-port LLDP indicator dot before the St column: match   mismatch   missing (when "Flag missing neighbors" is on)   missing (when flag is off)
  • St column — when devstatus data is available, the port dot reflects live interface operational state: connected   notconnect   errdisabled   other

Card indicators survive topology redraws — re-rendered automatically after each refresh.

LLDP Audit Integration

The LLDP Audit row in the modal controls how LLDP results feed into Audit mode:

  • Mismatches always flagged — any LLDP mismatch is automatically raised as a red error in Audit mode. This cannot be disabled.
  • Flag missing neighbors (checkbox) — when on, ports with no LLDP neighbor turn red and appear in Audit. Turn this off for early-stage labs where some interfaces are intentionally dark (not all cables patched yet).

Config Push

Config Push sends the generated EOS configuration directly to lab devices via SSH — no manual copy-paste required. It uses an EOS configure session for a safe, atomic commit and runs a post-push verification diff to confirm the device is in the desired state.

Requirements
  • Device Bridge must be running (python3 device_bridge.py) with METHOD = "ssh" or METHOD = "eapi"
  • Each device needs an IP or hostname — set in the Device Bridge modal (management IPs from the sheet are auto-detected)

Single-device push — from the copy strip or config modal:

  1. Select a device or link, then click Push on the strip. TopoAssist fetches the config (from cache if available, otherwise from the sheet) and runs a dry-run preview.
  2. A diff modal appears showing what will change. Review, then click Commit.
  3. The modal switches to a live phase tracker: Phase 1: Dry-run ✓ → Phase 2: Commit (updates live, retries shown) → Phase 3: Verify (confirms device matches). Green = in sync, amber = residual diff shown inline, red = failed.
  4. If nothing changed, an Already in Sync modal appears and auto-closes after 5 s.

Push All — from the Device Bridge modal:

  1. Open the Device Bridge modal and confirm the bridge is running (green dot).
  2. Click Push All. No prior config generation needed — configs are fetched automatically per device (cache hit = instant, cache miss = fetched from the sheet).
  3. Up to 3 devices are pushed concurrently. Each row shows live status: fetching config… → pushing… → verifying… → result.
  4. Results: Push successful, no config diff (green) = committed and verified clean. Amber = committed but residual diff or verify check failed. Red = push failed.
  5. Failed attempts retry up to 3 times automatically before reporting an error.
Note: Config push is supported for ssh and eapi transports only. The rest and gnmi transports are read-only.

💡 Tips & Notes

  • The topology auto-refreshes when sheet data changes (configurable polling interval via the Refresh settings).
  • Layout preferences (card width, gap, theme, sort mode, node positions) are saved in browser localStorage per device.
  • In Flexible mode, drag nodes to arrange the topology. Positions are saved automatically.
  • In Simple mode, use Group Rects to annotate logical groups (PODs, tiers). Click an existing rect to edit its label, color, and text position.
  • The Rename Sheet option (Sheet Manager > Change Sheet Name) safely renames the working tab and updates all internal references.
  • Non-Arista devices (IXIA, Spirent) appear in the topology for cabling context but no EOS config is generated for them.
  • Card Collapse — double-click any device's header bar to collapse it to just its title row. Cables to/from that device are hidden while collapsed. Double-click again to expand. Single click still selects the device. Collapsed state is saved in browser localStorage per device.
  • Po Group Highlight — clicking any port that has a po_ value set will highlight all ports across the topology sharing the same Port-Channel number with an amber outline. Press Escape to clear.
  • Role Pill Filter — click a role badge (e.g. SPINE, LEAF) on any device header to filter the topology to devices with that role. Click the same badge again to clear the filter.
  • Config Age — the copy/view toolbar shows how long ago each device's config was fetched (e.g. "just now", "3m ago"). The age refreshes automatically every 30 seconds.
