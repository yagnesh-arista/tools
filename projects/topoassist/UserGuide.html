📦 Installation

TopoAssist runs as a Google Apps Script bound to a Google Sheet. Choose the method that suits you — template copy (fastest), clasp (recommended for developers), or manual copy-paste (no tools required).

Option A — Template copy (fastest, 1 click)

Step 1 — Make a copy of the template sheet

Click the link below. Google Drive will prompt you to make a copy — click Make a copy. The new sheet is immediately ready with TopoAssist installed.

📋 Make a copy of the TopoAssist template

The copy is owned by your Google account — no sharing permissions are inherited from the original.

Step 2 — Grant permissions
  1. Open TopoAssist > Show Topology — Google will ask to authorize the script.
  2. Click Review permissions → choose your account → Allow.
  3. The sidebar opens and TopoAssist is ready to use.

Option B — clasp (recommended for developers)

Step 1 — Get the source

Clone or download the public repository:

git clone https://github.com/yagnesh-arista/tools.git
cd tools/projects/topoassist
Step 2 — Create a Google Sheet & Apps Script project
  1. Open sheets.new to create a new Google Sheet (or use an existing one).
  2. Go to Extensions > Apps Script — this opens a bound script project.
  3. In the Apps Script editor, click Project Settings (gear icon) and copy your Script ID.
Step 3 — Install clasp & authenticate 
npm install -g @google/clasp
clasp login --no-localhost

Clasp prints a Google OAuth URL — open it in a browser, sign in with the Google account that owns the sheet, then paste the redirect URL back into the terminal. Use --no-localhost so the flow works on remote/SSH machines where localhost:8888 is not reachable.

Step 4 — Point clasp at your script & push

Edit .clasp.json in the topoassist/ folder — replace the scriptId with yours:

{"scriptId": "YOUR_SCRIPT_ID_HERE", "rootDir": "."}

Then push all files:

clasp push --force
Step 5 — Run the initialiser
  1. Back in the Apps Script editor, select the onOpen function and click Run once to grant permissions.
  2. Return to your Google Sheet — a TopoAssist menu will appear in the menu bar.
  3. Open TopoAssist > Show Topology to launch the sidebar.

Device Bridge (optional — for live device checks)

Step 1 — Get the bridge script

Once TopoAssist is installed, open the sidebar, click the Device Bridge button (shield icon) and click Download Bridge. This saves device_bridge.py to your Downloads folder, always in sync with the installed version.

Alternatively download directly from the repo:

curl -O https://raw.githubusercontent.com/yagnesh-arista/tools/main/projects/topoassist/device_bridge.py
Step 2 — Requirements

Python 3.8+ is required. The bridge uses only standard library modules for SSH/eAPI/REST modes — no pip installs needed. For gNMI only:

pip install pygnmi
Step 3 — Configure & run

Edit the config block at the top of device_bridge.py:

  • METHOD — "ssh" (default), "eapi", "rest", or "gnmi"
  • JUMP_HOST — SSH jump server hostname; "" to reach devices directly
  • SSH_USER — device username (default "admin", passwordless SSH assumed)
python3 device_bridge.py

The bridge listens on http://localhost:8765. The Device Bridge modal shows a green dot when reachable.

🗺️ Typical Workflow

A typical TopoAssist project follows this sequence. Each step links to its detailed section below.

Step 1 — Add Devices TopoAssist > Device Manager

Add device names (Leaf1) or ranges (Leaf[1-4]). Set role (SPINE / LEAF / DUT…), model, rack, and mark MLAG pairs if applicable. Toggle visibility to hide devices from the canvas without deleting data. See Device Manager for full details.

Step 2 — Schema Setup (first time only) TopoAssist > Sheet Manager > Column & Formatting Manager

Choose which attribute columns to include (speed, xcvr, vrf, …) and click Save & Sync. For repeat projects with an existing sheet, this step is already done.

Step 3 — Fill the Sheet TopoAssist > Sheet View

Use the progressive column views to fill the sheet in stages: Interface → Transceiver → Speed → Mode → Port Channel → Vlan → SVI → IP → Show All. The SheetAssist Panel opens alongside and shows column-level hints. Sheet formatting (color-coding, conditional dimming) is applied automatically. See Sheet Filling for details.

Step 4 — Load Topology TopoAssist > Show Topology

Open the sidebar (or click Refresh if already open). The sidebar reads the sheet and draws the cable topology on the canvas.

Step 5 — Annotate with Group Rects

Switch to Simple (Holistic) view. Click the Group Rect button in the TOPO EDIT floating panel (bottom-right of the canvas) and drag to draw a labeled rectangle around a logical group — POD, tier, DC. See Group Rects.

Step 6 — Configure Technology Stack & Auto Config

Open the Tech Stack modal to enable GW, P2P, BGP/OSPF underlay, VXLAN/EVPN overlay, and MLAG. Open the IP Config modal to set base subnets, loopback base, BGP ASN base, and per-device GW overrides. Details in Configuration Generation.

Step 7 — Check Audit

Audit runs automatically when the topology loads. Design issues (VLAN mismatches, Po asymmetry, mode conflicts) are flagged on the canvas with a red badge count. Check the Audit tab in the status log for the full issue list. Details in Audit Engine.

Step 8 — Verify with Device Bridge

Open the Device Bridge modal and download device_bridge.py — then double-click to run it locally. Click Run Checks to overlay LLDP results on every port — green = match, amber = mismatch, red = missing. Details in Device Bridge.

Step 9 — Generate & Push Config

Click any cable or device to open the Copy Strip. Click View to preview EOS config or Push to send it via SSH in an EOS configure session. Review the diff modal, then click Commit.

Step 10 — Cleanup Orphans

After removing or renaming rows, open Device Bridge → Cleanup to find and remove stale __TA-tagged interfaces that are no longer in the topology.

🖥️ Device Manager

TopoAssist > Device Manager

Device Manager is the master list of every device in the topology. Each row creates a column group in the sheet and a node on the canvas.

Adding Devices

  • Enter names individually (Leaf1) or as a range (Leaf[1-4], Spine[01-02] for zero-padded). Ranges expand at save time.
  • Select Arista type for EOS devices — they get all attribute columns and config is generated. Select Non-Arista for traffic generators and third-party devices — they get only the int_ column (cabling context, no EOS config).
  • Drag rows to reorder devices — this affects sheet column order and changes Device IDs.

Editing Devices

All editable fields in Device Manager are inline — no separate edit mode or pencil column. Click any field to edit it directly in the row:

  • Device name — click the name cell and type a new name. The name propagates to column headers and canvas nodes on Save.
  • DNS name — click the DNS cell and type the hostname or IP the Device Bridge should use to reach this device. Leave blank to fall back to the device name.
  • Select fields (type, role, etc.) — click to open the dropdown and choose a new value.

Change indicators — edited fields turn green and show a small was: original label so you can see what changed before committing. Select fields gain a green border. All indicators are cleared after a successful Save.

Changes are pending until you click Save. Click Cancel to revert everything to the last-saved state. Closing the modal without saving discards all pending edits.

Device ID (DID)

Each Arista device is assigned a sequential Device ID — its position among all Arista devices in the sheet (1-based). DID drives:

  • Loopback0 IP — id.id.id.id/32 where id = DID + lo_base
  • BGP ASN — asn_base + DID
  • VARP physical IP last octet — same id value
Inserting a device mid-project shifts the DID of every Arista device below it. After any insert, run Regenerate All Configs from the Device Bridge modal, then re-push all affected devices. TopoAssist detects a DID shift at push time and shows a yellow banner as a reminder.

MLAG Pairing

Select two Arista devices in Device Manager and click Set MLAG Pair. Physical links between the pair are automatically tagged as Peer-Links. MLAG is declared explicitly — TopoAssist never infers it from Port-Channel counts.

Non-EOS Toggle

Any device can be toggled between EOS and Non-EOS by clicking the EOS badge in its Device Manager row. Devices with the IXIA or SPIRENT role are Non-EOS automatically. Non-EOS devices appear in the topology for cabling context but no config is generated. Changes take effect after Save + Force Sync.

Model, Rack & DNS Name

  • Model — platform string (e.g. 7050CX3-32S). Auto-populated from a Device Bridge check; manual entry overrides. Shown as the first pill under the node in Simple view.
  • Rack — physical location (e.g. dm1-rack104-tb38). Manual only. Appears in the Cabling Report.
  • DNS name — hostname or IP address the Device Bridge uses to reach this device (e.g. leaf1.lab.example.com or 192.0.2.11). When blank, the Device Bridge falls back to the device name. Set this when the lab management IP or DNS name differs from the topology name.

Visibility

Click the eye icon on any device row to hide it from the topology canvas. The device row and all sheet data are retained — only the visual node and its cables are suppressed. Useful for hiding maintenance or out-of-scope devices during a demo.

📋 Sheet Filling

The Google Sheet is the primary data source. Filling it in stages — using the column visibility views as a guide — keeps the workflow structured and reduces errors.

Progressive Column Views

TopoAssist > Sheet View

Work through the views in order to fill the sheet without being overwhelmed by all columns at once:

  1. L1: Interface — int_ columns only. Fill in which port on each device connects to which port on another device.
  2. L1: Transceiver — adds xcvr_type_. Record the transceiver type installed in each port.
  3. L1: Speed — adds et_speed_ and xcvr_speed_. Color-coding immediately highlights speed mismatches between port speed and transceiver speed.
  4. L1.5: Mode — int_ + sp_mode_. Mode drives config generation and canvas color-coding. See the mode pills in the Configuration Generation section for valid values.
  5. L1.5: Port Channel — adds po_. Assign Po numbers for trunk and Po-mode ports.
  6. L2/L3: Vlan — adds vlan_. Fill allowed/access VLANs and dot1q encap VLANs for sub-interfaces.
  7. L2/L3: SVI — adds svi_vlan_. Specify which VLANs get a routed SVI (Switched Virtual Interface) (all, a list, or blank).
  8. L2/L3: IP — adds ip_type_. Set the IP role for each port (p2p, gw, etc.).
  9. Show All — unhides all columns for a final review before generating config.

SheetAssist Panel

Opening any Sheet View also opens the SheetAssist Panel alongside the sheet. It shows column-level hints, valid values, and quick-fill guidance for the columns currently visible. Use it while filling to avoid typos and find correct mode, xcvr, and speed values.

Sheet Formatting

Connection background colors, conditional dimming, and speed tier colors are applied automatically as you fill the sheet. If formatting drifts (e.g. after adding a column or running a schema change), re-sync it:

TopoAssist > Sheet Manager > Column & Formatting Manager

Check the Apply GSheet Coloring checkbox before clicking Save & Sync.

Connecting Interfaces

Option A: Spreadsheet Matrix (fastest)
Find the column for the remote device (e.g. int_Leaf2). In the row for the local device (e.g. Leaf1), type the port name (e.g. Et1). The topology refreshes automatically.
Option B: Visual Edit Mode
Enable Edit Mode in the toolbar. Click a red dot (disconnected port) on one device, then click a red dot on another device to cable them.

🎨 Visual Color Coding

Sheet — Connection Backgrounds

  • Each cable pair gets a unique pastel color — both endpoints share the same shade so you can follow a cable across the sheet at a glance.
  • MLAG Port-Channels get an additional black medium border.
  • Peer-Link cables use a distinct color.
  • Orphan ports (no paired connection in this row) remain white.

Sheet — Conditional Dimming

  • Member port (Et with a Po filled): text goes italic/muted — connection color still shows through.
  • VLANs column accepts standard formats (10, 10,20, 10-20) plus an optional nv<N> token for the trunk native VLAN — e.g. 10,20,nv100 sets allowed VLANs 10 and 20 with native VLAN 100. Only one nv<N> token per cell.
  • SVI VLANs column is grayed when mode is l3-* (SVI is only valid on L2 modes). Value: all = SVI for every VLAN in the VLANs column (including native); 10 or 10,20 = SVI only for those VLANs; nv100 = SVI for native VLAN 100; blank = no SVI.
  • Po column is grayed when mode is l2-et-* or l3-et-* (no Po on direct Ethernet modes).
  • IP Type column is grayed when mode is l2-* and SVI is empty.

Sheet — Speed Colors (et_speed_ and xcvr_speed_)

Both the et_speed_ (EOS interface speed) and xcvr_speed_ (transceiver speed) columns use the same speed-tier palette — so mismatches between port speed and module speed are immediately visible:

1.6T 800G 400G 200G 100G 50G 40G 25G 10G 1G

Conditional rules on xcvr_speed_:

  • Amber — xcvr_speed_ equals et_speed_: the transceiver is likely a breakout module but xcvr_speed was set to per-lane speed instead of the aggregate. Example: QSFP28 in 4×25G mode with et_speed=25g and xcvr_speed=25g — xcvr_speed should be 100g.
  • Pale orange — xcvr_speed_ is empty AND the xcvr_type_ native speed does not match et_speed_ (breakout detected). Fill in xcvr_speed_ with the aggregate module speed to enable correct cable-group detection in the cabling report.

Sheet — Transceiver Type Colors (xcvr_type_)

The xcvr_type_ column is color-coded by form factor, using the same palette as the speed columns where the native speeds align:

OSFP QSFP-DD QSFP56 QSFP28 / QSFP100 / DSFP QSFP+ SFP28 / SFP25 SFP+ SFP (1G) BASE-T

The palette intentionally aligns with speed colors: a QSFP28 (100G native) and a 100G speed cell both show green (#dcfce7). This makes it easy to spot when a module is running in breakout mode — xcvr_type shows green (100G) but et_speed shows sky blue (25G).

SPINE LEAF HARNESS TRANSIT DUT IXIA SPIRENT

Loopback & VTEP (Virtual Tunnel Endpoint) Pinned Rows

Every Arista device card shows pinned rows above the physical port list for logical interfaces — Loopback and Vx1. These rows have no cabling association and never trigger audit checks. Click any pinned row to view, copy, or push its generated config.

Loopback rows (blue)

  • Lo0 — present on every Arista device. Carries the unique device IP (id.id.id.id/32, where id = sheetIndex + lo_base). Used as BGP/OSPF router-id and underlay peer address. For standalone VTEPs, also used as vxlan source-interface.
  • Lo1 — present on MLAG VTEP devices only. Carries the shared MLAG pair IP (min.min.max.max/32, same on both peers). Used as vxlan mlag source-interface Loopback1 so remote VTEPs see the pair as a single VTEP.
  • Cat column: LO pill.

Vx1 row (teal) — VTEP devices only

For LEAF devices with VXLAN configured (one or more Vx1 rows in the sheet), a pinned Vx1 summary row appears below the loopback rows.

  • Et column: Vx1 — shown in teal to distinguish it from physical ports.
  • Mode column: AP VLAN list in compressed range format (e.g. 10,20-25,100) — collected from all Vx1 rows for that device. Hover to see the full list if truncated.
  • Cat column: VTEP pill.
  • Click to open the interface Vxlan1 config block — view, copy, or push it independently.

🔧 Sheet & Schema Management

Column Visibility Views

TopoAssist > Sheet View

Quickly focus the sheet on specific data without losing any columns:

  • Show All — unhide everything.
  • L1: Interface — int_ columns only.
  • L1: Transceiver — int_ + xcvr_type_.
  • L1: Speed — int_ + xcvr_type_ + et_speed_ + xcvr_speed_. Cross-check transceiver type against port and module speeds.
  • L1.5: Mode — int_ + sp_mode_.
  • L1.5: Port Channel — int_ + sp_mode_ + po_.
  • L2/L3: Vlan — int_ + sp_mode_ + po_ + vlan_.
  • L2/L3: SVI — adds svi_vlan_.
  • L2/L3: IP — adds ip_type_.
  • Custom View — pick any column combination and save it.

Column & Schema Manager

TopoAssist > Sheet Manager > Column & Formatting Manager
  • Add, remove, or reorder attribute columns (affects all devices).
  • Customise dropdown options for any column (e.g. add a new speed value).
  • Save & Sync rebuilds the sheet with the new schema — existing data is preserved. If orphaned columns are detected (columns no longer in the schema), the Force Sync button turns yellow. A Delete & Sync / Cancel confirmation is shown before data is removed — the schema is the source of truth and orphaned columns cannot be kept.
  • Schema prefix guard — adding a field key that is a prefix of an existing key (e.g. adding xcvr when xcvr_speed exists) is blocked at both the client and server. This prevents silent data loss during sync.
  • Apply GSheet Coloring checkbox — re-applies connection backgrounds and conditional rules during sync.
  • Debug log in F12 checkbox — enables verbose debug output to the browser console (F12 → Console). Useful for diagnosing fetch, cache, and LLDP issues. Off by default; persists across reloads and browser restarts via localStorage. Toggle off when not needed.
Note: After editing conditional format rules, always run Save & Sync from the Schema Manager to push the changes to the sheet.

Schema Audit (Check Sync)

The Check Sync button in the Schema Manager footer compares the live sheet columns against the Device Manager configuration and reports any drift. Run this after adding or removing devices, or after a manual column edit outside the schema manager.

  • Missing columns — a column defined in the schema is absent from the sheet. The sheet is out of sync and config generation may silently skip fields. Fix by running Save & Sync.
  • Extra columns — a column exists in the sheet that is not in the schema. These are orphaned columns — they will be removed on the next Save & Sync (confirmation required).
  • Per-VLAN VRF issues — when vrf_ contains a comma-separated list, the count must match the VLAN count exactly. Errors reported:
    • VRF count ≠ VLAN count (e.g. 3 VLANs, 2 VRF entries)
    • VLAN range (10-20) combined with multiple VRF values — expand the range to an explicit list first
    • Port mode is not sub-interface or trunk SVI — per-VLAN VRF has no effect on access or pure-L2 ports
    • Trunk port has multiple VRF entries but svi_vlan_ is blank (per-VLAN VRF only applies to SVIs)
Check Sync is read-only — it reports issues but does not modify the sheet. Use Save & Sync to repair missing or orphaned columns.

Live Sheet Validation

TopoAssist automatically validates and cleans sheet data on every manual edit — no explicit action required. Three layers run in sequence:

  • onEdit (real-time) — fires on every manual cell change:
    • Column A guard — Column A is reserved for internal row visibility tracking. Any accidental edit is automatically reverted; if column A is deleted entirely, TopoAssist restores it automatically with a notification.
    • Format validation — VLAN format (10, 10,20, 10-20, nv100, or blank) and PO format (Po10 pattern) are checked on edit with a toast warning. Interface type (int_) is not restricted on edit — unsupported prefixes are flagged in Audit mode only.
    • Cell background rules — po_ is grayed when the mode is Ethernet (N/A); svi_vlan_ is grayed when the mode is L3 (N/A). Applied automatically after each edit.
  • validateAndCleanData (field-level) — called from onEdit for each changed cell:
    • Normalises PO values (Port-Channel10 → Po10)
    • Validates VLAN syntax — rejects values that don't match the allowed pattern
  • runFullSheetCleanup (background) — runs automatically on background fetch and can also be triggered manually:
    • Interface name normalisation: Ethernet1 → Et1, Port-Channel10 → Po10
    • SVI VLAN standardisation: true / 1 / yes / active → all
    • Returns a count of cells changed; no-op if the sheet is already clean
These functions clean format issues only. Configuration logic errors (mode conflicts, VLAN mismatches, duplicate interfaces) are flagged by the Audit Engine — they are intentionally not auto-corrected.

⬛ Group Rects

Group Rects let you draw labeled rectangles over the Simple (Holistic) topology view to mark logical groups — PODs, tiers, data centers, or any custom grouping. They are purely visual annotations stored in browser localStorage.

  1. Switch to Simple view (◎ button in the secondary row of the toolbar).
  2. Click the Group Rect button (dashed square icon) in the TOPO EDIT floating panel at the bottom-right of the canvas to enter draw mode.
  3. Click and drag on the canvas to draw a rectangle over the devices you want to group.
  4. A label input appears — type a name (e.g. POD-A, DC-West) and press Enter.
  5. Click an existing rect to edit its label, text position (inside / outside), or background color. Drag it to reposition. Press Delete or the trash icon to remove it.
Group Rect positions and labels are saved in browser localStorage per topology. They persist across page refreshes but are local to your browser — they are not written to the sheet.

⚙️ Configuration Generation

1. Interface Naming

The system validates and normalises Arista EOS interface names. Not all interface types are entered in the sheet — the table below shows where each is configured:

Abbrev Type How it is configured
Et Ethernet Enter in the int_ column (e.g. Et1, Et5/1)
Vx VXLAN (VTEP) Enter Vx1 in int_ — the VTEP row; VXLAN block is auto-generated from VLAN/VNI (Virtual Network Identifier) settings
Po Port-Channel Po number goes in the po_ column (not int_); member ports are listed in int_
Lo Loopback Auto-generated — derived from IP Config settings (loopback base + device DID formula); not entered in int_
Vl VLAN SVI Auto-generated — built from the vlan_, svi_vlan_, and ip_type_ columns; not entered in int_
Ma Management Not configured by TopoAssist — management addressing is outside the lab topology scope
Only Et and Vx1 are entered in the int_ column. Putting Lo, Vl, Ma, or Po in int_ will trigger an audit error.

2. Device Model & Rack

Each device in Device Manager has optional Model and Rack fields (shown as a compact sub-row below the device name).

4. IP Addressing (VLAN-Derived)

Middle octets are derived from the VLAN ID:

  • VLAN 1050 → octets .10.50
  • Last octet = Device DID (sequential Arista device index, set in Device Manager).
IPv6 uses the same pattern as IPv4 for lab convenience — this intentionally deviates from strict Arista EOS syntax.

IP Configuration Modal

Open with the toolbar button. Configure global GW and P2P base addresses, loopback and VNI bases, BGP ASN base, bridge MAC, and per-device GW overrides. Changes take effect on the next config generation; saving also invalidates all cached configs so they are regenerated on next view.

GW Address Formula

Gateway SVIs are addressed using four configurable fields — First, Last, and Mask for each of IPv4 and IPv6 — combined with the VLAN-derived octets.

Standard prefix (IPv4 ≥ /24, IPv6 ≥ /64):

  • IPv4: {First}.{oct2}.{oct3}.{Last}/{Mask} — e.g. first=100, VLAN 1520 → 100.15.20.1/24
  • IPv6: {First}:{oct2}:{oct3}::{Last}/{Mask} — e.g. first=100, VLAN 1520 → 100:15:20::1/64

Wide prefix (IPv4 < /24, IPv6 < /64) — first field becomes a numeric prefix:

  • IPv4: {First}{oct2}.{oct3}.0.{Last}/{Mask} — e.g. first=1, VLAN 4050 → 140.50.0.1/16
  • IPv6: {First}{oct2}:{oct3}:0::{Last}/{Mask} — e.g. first=1, VLAN 4050 → 140:50:0::1/32
In wide-prefix mode the First field is locked to 1 (the only universally valid prefix — larger digits overflow 255 for VLANs ≥ 1000). The live formula examples in the modal update as you type to show the resulting address for a representative VLAN.

Per-Device GW Override (Advanced)

The GW Auto Config accordion in the IP Config modal exposes a per-device override table. Each Arista device gets six fields: v4 First, v4 Last, v4 Mask, and the IPv6 equivalents. Fields left empty inherit the global GW values (shown as grey italic placeholder text).

How overrides work:

  • Any non-empty field replaces the global value for that device only.
  • Empty fields and grey-italic (hint) fields always inherit from global. Clearing a field back to empty re-inherits.
  • To enter an explicit value: click the field (the grey hint text clears automatically), type the value, and tab away. The field stays white to confirm the override is active.
  • To reset a field back to global: clear it and click away — the hint re-appears.

Wide-prefix mode in per-device overrides:

  • When the effective mask for a device is wide (either the per-device override mask, or the global mask if no override), the First field for that device is locked to 1 and disabled.
  • If the device had an explicit First override before the mask went wide, the value is preserved internally and restored when the mask returns to a standard prefix.
  • Hint-mode First fields (no override) are also forced to 1 in wide mode and cleared on exit.
Overrides are stored in DocumentProperties (per-spreadsheet) and applied at config generation time — they do not affect the sheet rows.

MLAG Constraint

MLAG peer pairs must share the same GW First octet (they share a common anycast or virtual-router IP). TopoAssist warns on save if the per-device First overrides for an MLAG pair differ, and highlights mismatched fields amber in the table. You can proceed past the warning, but the generated config will have a mismatch that EOS will reject at push time.

7. Technology Stack

Configure interface types and routing protocols via the toolbar button. These flags control which config blocks are included in the generated output.

  • GATEWAY (GW) INTERFACE — two independent checkboxes: IPv4 and IPv6. Controls whether gateway addressing is emitted on SVI, sub-interface, and L3 routed ports. The gateway mechanism (Anycast GW vs VARP) is configured in the EVPN VGW section below.
  • POINT-TO-POINT (P2P) INTERFACE (5-mode radio) — sets the addressing mode for point-to-point underlay links:
    • IPv4 — ip address only
    • IPv6 — ipv6 address only (explicit, no IPv4)
    • IPv6 Unnum — ipv6 enable only (link-local for RFC 5549 BGP, no IPv4)
    • IPv4 + IPv6 — both ip address and ipv6 address
    • IPv4 + IPv6 Unnum — ip address + ipv6 enable (link-local IPv6)
  • PROTOCOL UNDERLAY — two protocol rows with variant checkboxes:
    • BGP — IPv4 · IPv6 · Unnumbered · RFC5549
    • OSPF — v2/IPv4 · v3/IPv6 · Unnumbered
    Checkboxes incompatible with the selected P2P mode are greyed out and automatically unchecked.
  • PROTOCOL OVERLAY — two protocol rows:
    • VXLAN — IPv4 VTEP · IPv6 VTEP
    • EVPN — IPv4 · IPv6 address families, plus two sub-sections:
      • SERVICE — Per-VLAN (separate EVPN instance per VLAN, RT = VLAN:VLAN) or Bundle (single EVPN_VLAN_AWARE_BUNDLE, RT = ASN_BASE:1 — must be identical on all VTEPs)
      • VGW — gateway mechanism: Anycast GW (emits ip address virtual, same IP on all VTEPs) or VARP (unique physical ip address per device + shared ip virtual-router address). When VARP is selected, a Virtual Router MAC input appears — this MAC must be identical on all switches in the domain.
  • Validation errors — the modal blocks Save and shows an error banner if an invalid combination is chosen (e.g. IPv6 GW with no IPv6 routing protocol, VXLAN without an underlay, EVPN without BGP).

8. Snake Test (L2 / L3)

A snake test chains switch ports with external loopback cables to measure forwarding throughput end-to-end. A traffic generator (IXIA, Spirent, etc.) connects two ports to the switch — EP1 and EP2. The VRF chain routes traffic bidirectionally: EP1 → chain → EP2 (forward) and EP2 → chain → EP1 (reverse). Both directions use the same loopback cables and the same static ARP trick, so a single wiring is enough to measure full-duplex throughput.

Each snake pair occupies one row. Open the Edit modal with the same device on both sides (devA = devB) — TopoAssist detects this and writes to both int_ and snake_int_ columns automatically. You can also fill the sheet directly:

Snake pairs appear as orange arcs looping off the right side of the device node on the canvas.

L3 VRF Chain — What Gets Generated

For each L3 snake pair, TopoAssist auto-generates across four config sections:

  • 001_SYSTEM — vrf instance SNAKE_Et2 + ip routing vrf SNAKE_Et2 (one entry per port; no ipv6 unicast-routing in snake VRFs).
  • 050_EtX — interface block per port: vrf SNAKE_EtX, ip address {base}.oct2.oct3.1/24, no shutdown. No OSPF (cannot form adjacency with yourself). Both the primary and secondary get their own 050_ block.
  • 050_TRAFFIC_SNAKE_EP1_L3 / 050_TRAFFIC_SNAKE_EP2_L3 — regular P2P interface blocks, placed in the first/last snake VRF respectively (auto-assigned from desc_).
  • 080_SNAKE — static ARP + bidirectional route chain. Two bridge-MAC ARP entries per pair. Each EP field accepts comma-separated CIDRs; one ip route is emitted per subnet per VRF hop. Forward routes use EP2 Subnets (EP1→EP2); reverse routes use EP1 Subnets (EP2→EP1). Legacy 3-octet prefix (e.g. 10.99.99) is auto-expanded to 10.99.99.0/24. Empty field = no routes for that direction (ARP skeleton only).
  • 081_SNAKE_PBR — TTL reset via Policy-Based Routing: a shared ip access-list SNAKE_TTL_MATCH (permit ip any any) + route-map SNAKE_SET_TTL permit 10 (set ip ttl 64), applied with ip policy route-map SNAKE_SET_TTL on every snake primary and secondary interface.
  • 082_SNAKE_TRAFFIC_POLICY — TTL reset via Traffic Policy: a traffic-policy SNAKE_TTL_POLICY block matching all IPv4 and setting set ip ttl 64, applied traffic-policy input/output on every snake interface. Use either this or PBR — both reset TTL to 64 so the switch does not drop forwarded frames that have exhausted their hop count.

TTL Reset — Why It Matters

Each loopback hop in the snake chain decrements the IP TTL by 1. A 64-pair snake test consumes 64 hops — a standard TTL-64 frame would drop at the last pair. Both generated sections reset TTL to 64 on ingress so every hop starts with a full TTL. Choose one method to apply:

  • PBR (081_SNAKE_PBR) — simpler; no license required. Works on all platforms. The route-map fires before the routing decision; all IPv4 traffic entering a snake port gets TTL reset.
  • Traffic Policy (082_SNAKE_TRAFFIC_POLICY) — hardware-based; matches at line rate. Preferred for high-rate testing if Traffic Policy is available on the platform.
Apply only one method — both generate valid config but deploying both simultaneously is redundant. Paste the preferred section into the device and skip the other.

End-to-End Traffic Path (L3)

Forward (EP1 → EP2):

  1. Traffic generator sends frames destined to the EP2 subnet (e.g. 172.16.1.98, configured via EP2 Subnet field) into TRAFFIC_SNAKE_EP1_L3.
  2. EP1 port shares VRF SNAKE_Et2 (first primary). The static route points to 200.2.0.2; bridge-MAC ARP maps that IP to the switch's own MAC → frame exits Et2.
  3. Loopback cable carries the frame to Et3. Switch accepts it (DMAC = own MAC). In VRF SNAKE_Et3, the route uses egress-vrf SNAKE_Et4 → frame exits Et4.
  4. Loopback to Et5. In VRF SNAKE_Et5, the forward terminal route directs the frame to EP2's NH (static ARP if indirect, dynamic ARP if directly connected).
  5. Traffic generator receives the frame on EP2.

Reverse (EP2 → EP1):

  1. Traffic generator sends frames destined to the EP1 subnet (e.g. 10.100.1.99, configured via EP1 Subnet field) into TRAFFIC_SNAKE_EP2_L3.
  2. EP2 port shares VRF SNAKE_Et5 (last secondary). The reverse route points to 200.4.0.2 → frame exits Et5 back through the cable to Et4.
  3. In VRF SNAKE_Et4, the reverse route uses egress-vrf SNAKE_Et3 → frame exits Et4's cable to Et3.
  4. In VRF SNAKE_Et3, the reverse route points to 200.2.0.2 → exits Et3 back to Et2.
  5. In VRF SNAKE_Et2, the reverse terminal route exits to EP1's NH. Traffic generator receives the frame on EP1.

Manual Steps Required

  1. Loopback cables — one patch cable per snake pair (Et2↔Et3, Et4↔Et5, …).
  2. Bridge MAC — set in the IP Config toolbar button → Snake Test tab. Auto-detected if Device Bridge is running; otherwise read from EOS with show version | grep "System MAC".
  3. EP1/EP2 NH + MAC (if indirect) — if the traffic generator is not directly cabled to the snake port (there is a router in between), set the NH IP and MAC for that endpoint. If directly connected, leave blank — the switch ARPs dynamically.
  4. EP1/EP2 Subnets (optional) — enter one or more comma-separated CIDRs (e.g. 10.1.1.0/24, 10.2.2.2/32, 10.3.0.0/16). Legacy 3-octet prefixes (e.g. 10.99.99) are also accepted. One ip route is generated per subnet at every VRF hop in the chain. EP2 Subnets drive forward routes (EP1→EP2); EP1 Subnets drive reverse routes (EP2→EP1). Leave blank for no static routes in that direction.
  5. EP1 / EP2 descriptions — set desc_=TRAFFIC_SNAKE_EP1_L3 on one generator-connected port row and desc_=TRAFFIC_SNAKE_EP2_L3 on the other. The snake VRF chain is built relative to these two anchors.

Everything else — VRF declarations, interface configs, ARP entries, and the full bidirectional routing chain — is generated automatically from the sheet data.

9. Per-VLAN VRF

The vrf_ column supports assigning a different VRF to each VLAN on a trunk or sub-interface port by supplying a comma-separated VRF list that maps positionally to the VLAN list in vlan_.

Single VRF (legacy behavior)

A single value in vrf_ applies the same VRF to all VLANs on the port — unchanged from before:

vlan_ = 10,20,30    vrf_ = MGMT
→ all three VLANs placed in VRF MGMT

Per-VLAN VRF (comma list)

Provide one VRF name per VLAN, in the same order as vlan_:

vlan_ = 10,20,30    vrf_ = RED,BLUE,GREEN
→ VLAN 10 → VRF RED  |  VLAN 20 → VRF BLUE  |  VLAN 30 → VRF GREEN

Works on both:

  • Sub-interface mode (l3-routed-subint) — each sub-interface (EtX.10, EtX.20, …) gets its own vrf statement from the positional mapping.
  • Trunk SVI mode (l2-*-trunk + svi_vlan_) — each SVI (interface Vlan10, interface Vlan20, …) gets its own vrf statement from the positional mapping.

Constraints

  • No range expansion with per-VLAN VRF — ranges (10-20) expand to many VLANs; assigning individual VRFs to each would require listing all of them explicitly, which is ambiguous. The audit engine flags this combination as an error. Use explicit lists (10,11,12,…) when per-VLAN VRF is needed.
  • Count must match — the number of VRF values must equal the number of VLANs. A mismatch is flagged as an error. Trailing commas and blank entries are ignored.
  • Native VLAN excluded — the nv<N> token in vlan_ (e.g. 10,20,nv100) is not counted as a positional VLAN — it only sets the trunk native VLAN. VRF list length must match the non-native VLAN count only.
  • IXIA / snake VRFs take precedence — if the port has desc_=TRAFFIC_SNAKE_EP* or ip_type=anycast, the snake/anycast VRF overrides the per-VLAN mapping.
Per-VLAN VRF is most useful on uplink trunk ports where different customer VLANs need to be isolated into separate VRFs. For access ports or pure L2 trunks without SVIs, the vrf_ column has no effect.

Audit Engine

Single-click the Audit icon to open the Audit Checks settings modal — configure which of the 24 checks are enabled. Double-click the same icon to toggle audit mode on or off. The icon changes color to show audit state at a glance: dimmed when audit is disabled, green when enabled with no issues found, and red when enabled with issues present. A badge shows the error count when issues are found.

  • Unsupported Interface — int_ value does not start with Et (Ethernet) or Vx (VTEP). All Arista physical ports use the Et prefix regardless of speed; Vx1 is the only valid VTEP entry. Other types (Po, Lo, Ma, etc.) do not belong in int_.
  • Po Mismatch — both sides of a cable have po_ set but to different Port-Channel numbers. Both ports are flagged.
  • Po Asymmetry — one side of a cable has po_ set and the peer does not. Both ports are flagged.
  • VLAN Mismatch — the vlan_ values differ across a cable (Arista-only links).
  • Mode Incompatible — the sp_mode_ pair across a cable is not a supported combination.
  • Missing Po for Port-Channel mode — interface mode is po-based but po_ is blank.
  • Stale Po value on Ethernet mode — interface mode is et-based but po_ has a value.
  • Orphan Po (Arista only) — an Arista device port is in Po mode with a po_ value but no peer device appears on the same row. Indicates a single-device row design error — Po members must have a connected peer.
  • Duplicate Interface — the same int_ name appears more than once for the same device.
  • Speed Mismatch — both sides of a cable must have the same speed_ value.
  • VLAN Gateway Conflict — the same VLAN ID is assigned different gateway IPs across devices.
  • Native VLAN on non-trunk port — nv<N> token in vlan_ is only valid on l2-*-trunk modes. On access, L3, or sub-interface ports the token has no meaning and is silently ignored in config generation.
  • SVI VLAN Mismatch — a VLAN ID in svi_vlan_ is not present in vlan_. Config generation silently skips SVIs for VLANs not in the allowed trunk list — this check surfaces the mismatch so the VLAN can be added to vlan_ or removed from svi_vlan_. Applies to l2-* modes only; svi_vlan=all is exempt (all trunk VLANs are valid by definition).
  • Internal VLAN Conflict — a VLAN in the vlan_ column (including the native VLAN) matches an EOS-internal VLAN auto-allocated by the device (from show vlan internal). Requires a Device Bridge check to run first — the check is silently skipped when no bridge data is available. Configuring a port on an internally-allocated VLAN will fail on EOS. Can be suppressed via Internal VLAN in topology in the Audit Checks modal (single-click the Audit icon).
  • Per-VLAN VRF Issues — flags mismatches when vrf_ contains a comma-separated list. Conditions checked:
    • Error — VRF count does not match VLAN count (e.g. 3 VLANs but 2 VRF entries).
    • Error — VLAN range (10-20) combined with multiple VRF values — ranges expand to many VLANs making positional mapping ambiguous; use an explicit VLAN list instead.
    • Error — Port mode is not sub-interface or trunk SVI; per-VLAN VRF has no effect on access or pure-L2 ports.
    • Warn — Trunk port has multiple VRF values but no SVI VLANs configured (svi_vlan_ is blank) — per-VLAN VRF on a trunk only applies to SVIs.

Clicking a flagged port in the topology highlights it. Click the Audit icon again to toggle the mode off.

🔌 Device Bridge

The Device Bridge is a lightweight local Python server (device_bridge.py) that gives TopoAssist live read access to lab devices via SSH, eAPI, REST, or gNMI — without any cloud dependency. Its primary purpose is to reduce lab bring-up time by surfacing cabling errors, device identity issues, and interface state directly in the topology view.

Setup

Step 1 — Get the bridge script

Click the Device Bridge toolbar button (shield icon), then click Download Bridge in the modal footer. This saves device_bridge.py to your Downloads folder — it is always in sync with the current sidebar version.

Step 2 — Configure & run

Open device_bridge.py in a text editor and set the config at the top:

  • METHOD — transport to use: "ssh" (default), "eapi", "rest", or "gnmi"
  • JUMP_HOST — SSH jump server hostname (e.g. "bus-home"); set "" to SSH devices directly
  • SSH_USER — device login username (default "admin", passwordless)

Then run it in a terminal and keep it open:

python3 device_bridge.py

The bridge listens on http://localhost:8765. The modal shows a green dot when it is reachable.

Transport options:
ssh — CLI over SSH; requires OpenSSH + sshpass on the bridge machine. No pip deps.
eapi — Arista eAPI JSON-RPC over HTTPS (port 443); same output as SSH; single request per device. No pip deps.
rest — RESTCONF over HTTPS; OpenConfig YANG format. No pip deps. Requires EOS 4.22+.
gnmi — gRPC/gNMI; OpenConfig YANG format. Requires pip install pygnmi. Port 6030. EOS 4.22+.

Running a Check

Open the modal, enter or confirm the Host / IP for each device (defaults to device name for DNS-resolvable hosts), then click Run Checks. The bridge queries all devices in parallel. A spinner shows progress and updates as each sweep completes (LLDP done → LLDP + status done).

Auto-check on connect: when the bridge is running and the topology loads, LLDP + device status checks run automatically in the background. The status bar briefly shows the result. If extra cables are discovered (cables present on the devices but not in the sheet), the bridge toolbar icon gains an amber count badge — click the icon to open the modal and review them.

Device List — Status Dots

Each device row shows two indicator dots after the IP field:

SSH dot — green: bridge reached the device successfully.
SSH dot — red: SSH failed (wrong IP, auth issue, device unreachable).
SSH dot — gray: no IP configured for this device.
Hostname dot — green: device hostname matches topology name. Hover for confirmed hostname.
Hostname dot — amber: hostname mismatch — wrong device on that IP. Hover to see expected vs. actual name.
An amber hostname dot is a critical lab bring-up signal — it means the device responding on that IP is not the device you think it is. Check physical cabling and IP assignment before pushing config.

LLDP Results

Results are shown per cable, sorted by severity:

✓ Match — LLDP confirms the cable: both sides see the correct neighbor and port.
✗ Mismatch — LLDP sees a different neighbor or port than the topology expects. Detail line shows what was actually seen. Always flagged in Audit mode.
? Partial — one side was queried OK, the other was not (wrong IP or SSH failure on that device).
? Unknown — neither device was reachable; result is inconclusive.
— Skip — non-Arista device or non-Ethernet interface (LLDP check only covers Et↔Et Arista links).

Discovered — Not in Sheet

After a check, the bridge compares every LLDP-reported neighbor pair against the topology sheet. Cables that exist physically (both endpoints are topology devices) but have no matching row in the sheet appear in a Discovered — not in sheet section at the bottom of the modal, with a purple link count badge.

  • Each discovered cable is shown as a checkbox row: DevA:Et1 ↔ DevB:Et2.
  • Select one or more and click Add to Sheet to write them into the topology without leaving the modal.
  • If orphans are found, the bridge toolbar icon shows an amber count badge. The badge clears when you open the modal.
  • The status bar also shows an amber message: "Device Bridge checks complete — N extra cables found".

Topology Card Indicators

After a check, results appear directly on topology device cards — no need to keep the modal open:

  • EOS version line — shown under the device hostname (e.g. EOS 4.32.1F · 7050SX3-48YC12). Populated from show version.
  • LL column — per-port LLDP indicator dot before the St column: match   mismatch   missing (when Missing LLDP neighbor check is enabled in Audit Checks)   missing (when check is disabled)
  • St column — when devstatus data is available, the port dot reflects live interface operational state: connected   notconnect   errdisabled   other

Card indicators survive topology redraws — re-rendered automatically after each refresh.

Interface Operational State (St Column)

The St column on each topology card shows the live EOS interface operational state, fetched via show interfaces status during a bridge check. This is separate from LLDP — a port can be LLDP-matched but still be operationally down.

connected — link is up and the interface is operational. Physical signal detected, SFP present.
notconnect — interface is enabled but no physical signal detected. Common causes: unplugged cable, missing or faulty SFP, peer port is down.
errdisabled — EOS has administratively disabled the port due to a fault condition (e.g. BPDU guard, storm control, LACP timeout, port security). The port will not recover automatically — it requires shutdown / no shutdown to clear, after the root cause is resolved.
disabled / other — interface is administratively shut, or the state string returned is unrecognised. Gray is also shown when no bridge data is available for that port.
Interface state data requires a bridge check with METHOD = "ssh" or "eapi". rest and gnmi transports do not expose interface operational state via the bridge.

Internal VLANs

After a bridge check (METHOD = "ssh" or "eapi" only), the bridge also runs show vlan internal on each device and stores the results in the topology database. EOS auto-allocates internal VLANs for L3 sub-interfaces, MLAG peer-links, VXLAN underlay, and other platform features — they are not user-configured and do not appear in any port column.

  • Global search — type a VLAN number in the search box. If that number is an EOS-internal VLAN on one or more devices, those device cards are shown (highlighted, same as a device name match). Port rows on other devices that carry the same VLAN number are shown normally alongside them.
  • VLAN trace — enter a VLAN in the trace box (e.g. v1025). If the VLAN is EOS-internal on any device, the VLAN legend shows an annotation: "VLAN 1025 — EOS internal on: leaf1, leaf2". The annotation appears even when no port carries that VLAN explicitly in the topology sheet.
Internal VLAN data is only available after a bridge check and is not persisted between sessions — re-run the check to refresh. METHOD = "rest" and "gnmi" do not support show vlan internal; those paths return no internal VLAN data.

Bridge / LLDP Audit Checks

The three Bridge/LLDP audit checks (LLDP neighbor mismatch, missing LLDP neighbor, internal VLAN in topology) are managed in the Audit Checks modal — single-click the Audit icon in the toolbar to open it. All checks default to on. Toggling Missing LLDP neighbor off leaves missing-port dots gray instead of red without suppressing them from the device card entirely.

Config Push

Config Push sends the generated EOS configuration directly to lab devices via SSH — no manual copy-paste required. It uses an EOS configure session for a safe, atomic commit and runs a post-push verification diff to confirm the device is in the desired state.

Requirements
  • Device Bridge must be running (python3 device_bridge.py) with METHOD = "ssh" or METHOD = "eapi"
  • Each device needs an IP or hostname — set in the Device Bridge modal (management IPs from the sheet are auto-detected)

Single-device push — from the copy strip or config modal:

  1. Select a device or link, then click Push on the strip. TopoAssist fetches the config (from cache if available, otherwise from the sheet) and runs a dry-run preview.
  2. A diff modal appears showing what will change. Review, then click Commit.
  3. The modal switches to a live phase tracker: Phase 1: Dry-run ✓ → Phase 2: Commit (updates live, retries shown) → Phase 3: Verify (confirms device matches). Green = in sync, amber = residual diff shown inline, red = failed.
  4. If nothing changed, an Already in Sync modal appears and auto-closes after 5 s.

Push All — from the Device Bridge modal:

  1. Open the Device Bridge modal and confirm the bridge is running (green dot).
  2. Click Push All. No prior config generation needed — configs are fetched automatically per device (cache hit = instant, cache miss = fetched from the sheet).
  3. Up to 3 devices are pushed concurrently. Each row shows live status: fetching config… → pushing… → verifying… → result.
  4. Results: Push successful, no config diff (green) = committed and verified clean. Amber = committed but residual diff or verify check failed. Red = push failed.
  5. Failed attempts retry up to 3 times automatically before reporting an error.
Note: Config push is supported for ssh and eapi transports only. The rest and gnmi transports are read-only.

What Gets Pushed

The config sent to the device is built in two layers:

Layer 1 — Generated config (View Config)
The full EOS config is generated from the sheet and is exactly what View Config shows. This is the single source of truth — device_bridge.py does not transform or rewrite it in any way. Vxlan cleanup commands (default vxlan vlan 1-4094 vni, default vxlan flood vtep) are already embedded in the generated config so that stale additive sub-commands are cleared before new mappings are pushed — they are visible in View Config and appear in the dry-run diff.

Layer 2 — Push-time additions (device-state-dependent)
Two sets of cleanup commands are prepended by device_bridge.py at push time because they require knowing the device's current running state — they cannot be pre-computed from the sheet alone:

  • Orphan cleanup (all 5 categories) — when a full-device push is triggered (not single-interface), the bridge runs the same detection logic as Cleanup across all five orphan types and prepends the removal commands before the new config in the same configure session. Every full push is self-cleaning: stale interfaces, BGP neighbors, VLANs, VRFs, and OSPF passives are all removed atomically.
  • BGP ASN change — if the config being pushed contains a router bgp <new-asn> block and the device is currently running a different ASN, the bridge prepends no router bgp <old-asn>. EOS cannot change the AS number in-place; the old process must be removed before the new one can be committed. The push modal shows a yellow warning banner when this applies.
Both push-time additions are always visible in the dry-run diff before you click Commit — so the complete picture of what will change on the device is shown upfront, even for commands that come from device state rather than the sheet.

Cleanup — Orphan Removal

Cleanup scans selected devices for __TA-tagged config that is no longer in the topology — interfaces, BGP neighbors, VLANs, VRFs, and OSPF passive overrides left over from deleted or renamed rows.

  1. Open Config Center (toolbar grid icon) and click the Cleanup nav item.
  2. Select target devices using the Targets strip, then click Clean-up Targets. The bridge queries each device and displays detected orphans grouped by category.
  3. Review the list and click Clean Selected or Clean All to push removal commands. Each category is handled differently:
Category Detection Removal command
Interface __TA-tagged iface in show interfaces description not present in new config default interface EtX (physical/Po) · no interface EtX.Y (sub-iface) · no interface VlanN (SVI)
BGP neighbor __TA-tagged neighbor in router bgp whose device name is not in the topology router bgp <ASN> / no neighbor <IP>
VLAN VLAN name contains __TA and the vlan N line is absent from new config; VLANs 4093/4094 skipped no vlan <N>
VRF VRF description contains __TA and the vrf instance block is absent from new config no vrf instance <NAME>
OSPF passive Stale no passive-interface EtX sub-command in router ospf/ospf3 referencing an orphaned interface default passive-interface EtX — removes the override entirely; port falls back to passive-interface default with no residue
OSPF passive cleanup is the reverse of the others — TopoAssist emits no passive-interface EtX on active ports to allow OSPF adjacency. When that port is deleted, the override must be cancelled with default passive-interface EtX. This removes the stale sub-command entirely, leaving passive-interface default to cover the interface with no residual config. Interface names are normalized automatically (Et/Po in topology vs. Ethernet/Port-Channel from EOS).

💡 Tips & Notes

  • The topology auto-refreshes when sheet data changes (configurable polling interval via the Refresh settings).
  • Layout preferences (card width, gap, theme, sort mode, node positions) are saved in browser localStorage per device.
  • In Flexible mode, drag nodes to arrange the topology. Positions are saved automatically.
  • In Simple mode, use Group Rects to annotate logical groups (PODs, tiers). Click an existing rect to edit its label, color, and text position.
  • The Rename Sheet option (TopoAssist > Sheet Manager > Change Sheet Name) safely renames the working tab and updates all internal references.
  • Non-Arista devices (IXIA, Spirent) appear in the topology for cabling context but no EOS config is generated for them.
  • Card Collapse — double-click any device's header bar to collapse it to just its title row. Cables to/from that device are hidden while collapsed. Double-click again to expand. Single click still selects the device. Collapsed state is saved in browser localStorage per device.
  • Card Resize — drag the right edge of any device card to adjust its width. Width is saved per card in browser localStorage and restored on the next load.
  • Po Group Highlight — clicking any port that has a po_ value set will highlight all ports across the topology sharing the same Port-Channel number with an amber outline. Press Escape to clear.
  • Role Pill Filter — click a role badge (e.g. SPINE, LEAF) on any device header to filter the topology to devices with that role. Click the same badge again to clear the filter.
  • Config Age — the copy/view toolbar shows how long ago each device's config was fetched (e.g. "just now", "3m ago"). The age refreshes automatically every 30 seconds.
